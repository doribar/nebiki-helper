# nebiki-photo-judge 引き継ぎメモ

このフォルダは、値引きヘルパー本体 ZIP に同梱されている写真判定サーバーです。
次の ChatGPT セッションでは、値引きヘルパー本体の `CHATGPT_HANDOFF.md` を先に読み、このファイルも必要に応じて確認してください。

## 役割
- 本体から写真を受け取り、写真セットとして保存する。
- OpenAI API で `多い / どちらでもない / 少ない` の参考判定を返す。
- 値引完了時に本体から送られる人間判定を保存する。
- 次回の同じエリア判定を使い、前回のエリア判定が弱かった / ちょうどよかった / 強かった可能性を付ける。

## 2026-05-01 の重要修正
前回フィードバック対象を、単なる「同じエリアの過去データ」ではなく、以下に限定した。

- 同じ `sessionDate`
- 同じ曜日基準
- 同じエリア
- 現在の値引時刻の直前時刻
- 人間判定済み
- 未フィードバック
- 現在より古い写真セット

これにより、`5月1日15時` の写真セットへ、別日や別時刻の判定で誤ってフィードバックが付く問題を避ける。

## 秘密情報の扱い
- `.env` は ZIP に含めない。
- 実写真や `data/photo-groups.jsonl` も ZIP に含めない。
- APIキーを含む ZIP を共有しない。

## 2026-05-02 追加メモ: 過去の適切例を使った写真判定方針

今後の写真判定は、今回写真を単独でAIに見せて判定するだけでなく、過去の「適切だった」写真セットと比較させる方向に寄せる。

最終形は、同じ曜日基準・同じ値引時刻・同じエリアから、次回フィードバックで適切だった以下の3種類をAIに見せ、今回写真がどれに一番近いかを判断させる。

- 適切だった「多い」例
- 適切だった「どちらでもない」例
- 適切だった「少ない」例

データが揃うまでは、適切だった過去例を指定件数だけ取得し、「過去判定を今回にも流用してよいか」「過去より多そうか/少なそうか」をAIに比較させる。
過去例の判定は「どちらでもない」に限定しない。「多い」や「少ない」でも、次回フィードバック上で適切だったなら比較対象にする。

過去例の取得件数は、写真1枚数ではなく写真セット数として扱う。

- 弁当・麺類エリア: 1件
- フライ・鶏惣菜エリア: 2件
- それ以外のエリア: 3件

理由:
- 弁当・麺類は1セット4枚で入力が重いため1件。
- フライ・鶏惣菜は1セット2枚なので2件。
- その他は写真数が少ない想定なので3件。

比較時にAIへ考慮させる点:

- 撮影角度や距離の違い
- 写真に写るエリア幅の違い
- 商品の積み重なり方
- 空きスペース
- 商品密度
- 広く薄く残っているのか、狭く厚く残っているのか

実装時は、エリア内区分名、例: 正面 / バラ側 / 寿司側 / 麺類 / フライ側 / 鶏惣菜側、も可能なら写真メタデータとして保存し、同じ区分同士を比較できるようにする。

## 2026-05-02 追加修正: 過去の適切例をAI判定に渡す暫定実装

今回、AI判定の入力を「今回写真のみ」から「過去の適切だった写真セット + 今回写真」に寄せた。

### 過去適切例の定義
過去に人間が `多い / どちらでもない / 少ない` のいずれかを選び、その次の値引で同じエリアが `どちらでもない` になったため、写真判定サーバー側で `feedbackFromNextJudge === "good"` が付いた写真セットを「適切例」として扱う。

### 選定条件
`src/selectExamples.js` で以下を満たすものを選ぶ。

- 同じエリア
- 同じ曜日基準
- 同じ値引時刻
- `humanJudge` が `多い / どちらでもない / 少ない` のいずれか
- `feedbackFromNextJudge === "good"`
- 写真パスが存在する

取得件数は写真1枚数ではなく写真セット数。

- `弁当・麺類`: 1セット
- `フライ・鶏惣菜`: 2セット
- その他: 3セット

### AIプロンプトの方針
`src/openaiJudge.js` で、AIに以下を明示する。

- 過去適切例の判定は `どちらでもない` に限らない。`多い` や `少ない` もある。
- 今回写真が過去適切例と同じ判定として扱えそうなら、その判定を参考にする。
- 今回の方が過去より残量・密度・積み上がりが多そうなら `多い` 寄りへ補正する。
- 今回の方が少なそうなら `少ない` 寄りへ補正する。
- 構図、距離、売場幅、商品の積み重なり、空きスペース、商品密度、広く薄い/狭く厚い残り方を考慮する。

### 環境変数
`.env.example` に以下を追加。

```env
MAX_EXAMPLE_PHOTOS_PER_GROUP=6
```

これは過去適切例1セットあたりAIへ渡す最大写真枚数。既定値6なので、弁当・麺類の4枚セットを切らない。

### 確認
`npm run check` PASS。

## 2026-05-02 本体側メモ: 写真が揃っているエリアだけAI判定対象

本体側で、全エリアの写真を揃えなくても「写真あり」で値引開始できるようにした。
写真判定サーバーへ送られるのは、本体側で必要写真がすべて揃っていると判断したエリアのみ。
撮影途中のエリアはAI判定サーバーへ送らない。

この変更は主に本体側 `src/components/screens/PhotoCaptureScreen.tsx` と `src/hooks/useNebikiApp.ts` の変更であり、写真判定サーバー側のAPI仕様変更はない。

## 2026-05-02 追加メモ: .env の現在運用
ユーザーは写真判定サーバーの `.env` を次の前提で運用している。

```env
HOST=0.0.0.0
OPENAI_MODEL=gpt-5.4-mini
```

`HOST=0.0.0.0` は、スマホや別端末から Tailscale / LAN 経由で `http://PCのTailscale IP:3000` のように接続するため。
`.env.example` もこの運用に合わせた。

注意:
- `.env` は秘密情報なので ZIP / Git に含めない。
- `.env.example` には `OPENAI_API_KEY=` のように空欄またはダミー値だけを置く。
- GitHub push protection に引っかかった場合は、commit から `.env` を削除し、APIキーは再発行する。
