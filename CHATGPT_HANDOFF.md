# ChatGPT 引き継ぎメモ

この zip は、**ChatGPT がこの会話で最後に生成した版** を正本として扱う前提の引き継ぎ用です。

## 次のセッションでの扱い
- ユーザーがこの zip をアップロードした場合、まずこの zip の中身を最新の正本として扱ってください。
- 同じ会話の中で ChatGPT がさらに新しい zip を生成した場合は、**その最後に生成した zip** を新しい正本として扱ってください。
- ただし、ユーザーが「こちらが最新」「この zip を基準にして」などと明示した場合は、ユーザー指定を優先してください。

## zip 名のルール
- ChatGPT が更新版を出力するときの zip 名は、**固定にしない**でください。
- zip 名は、その回の修正内容が分かるように毎回変えて構いません。
- 次のセッションでは、**zip 名ではなく、その時点で ChatGPT が最後に出した zip** を最新の正本として扱ってください。

## 実務上のルール
- 部分修正よりも、現在の正本を直接更新して次の zip を作る流れを優先します。
- セッションをまたぐときは、このファイルの内容を前提にして、ChatGPT が最後に出した zip を基準に作業を継続します。

## 保存まわりの方針
- 現在セッション、次回値引に影響する記録、前回天気記録は、できるだけ **同じ状態の塊** として扱います。
- localStorage を直接その場で増減させるのではなく、React state を更新してから保存する流れを優先します。
- 戻るや取り消しを直すときは、画面状態だけでなく **次回値引記録 / 前回天気記録** まで含めて巻き戻るかを確認してください。
- `npm run check:logic` には、戻る履歴の積み方・復元スナップショット・保存記録の同梱確認も含めてあります。戻るまわりを触ったら必ず回してください。

## 値引率補正の時間帯ルール
- 「曜日基準で補正しきれない分」の補正上限は時間帯で変わります。
- 強める側は、15時は `+5%` 上限、17時以降は `+10%` 上限です。
- 緩める側は、15時は `-10%` 上限、17時以降は `-5%` 上限です。
- ただし、30分〜1時間後の雨 `+10%` / 雪 `+20%` があるときは、この「補正しきれない分」は重ねません。
- 最終値引の点数計算では `±5% = ±1点`、`±10% = ±2点` です。

## 補足
- このファイルは運用メモです。アプリ本体の挙動には影響しません。

## StartScreen の初期値
- StartScreen の入力は、前回使った内容を初期値として引き継ぎます。
- ただし、手動固定していない曜日・時刻は、その時点の現在値に自動追従します。
- この挙動を触ったら、前回入力の保持とトップへ戻った直後の初期表示を確認してください。

## 補正ラベルの方針
- 補正の内訳ラベルは、できるだけ短く同じ粒度でそろえます。
- 例: `気温 16〜20度 -1段` / `風 5m以上 +1段` / `近い雨 +10%` / `後の雪 +2段`
- 文章調の説明は要約や補足に寄せ、内訳はラベル列として読む前提にします。

## 自動チェックの方針
- `npm run check:logic` には、ロジック単位の確認だけでなく、曜日・時刻つきの運用シナリオ型チェックも含めます。
- 水曜17時、日曜15時、金曜19時30分のような現場シナリオを増やす方向で維持してください。


## 時間別お天気入力
- StartScreen では、ウェザーニュースの1時間ごと表示を見ながら **15〜21時** の天気 / 気温 / 風速を入力します。
- 15時値引のときは、**15時と17時** の予報を比較します。
- 17時の気温が15時より **5度以上低い** ときは **曜日基準 +1段** です。
- さらに、上の条件を満たしたうえで **17時の風速が15時より強い** ときは **さらに +1段** です。
- この補正は通常の曜日基準補正に合流するので、**「曜日基準で補正しきれない分」** の +5% / +10% 判定にも影響します。
- 17時以降は、同じ時間別入力をそのまま参照するので、15時で入れた 17時予報は初期値として自然に引き継がれます。


## 2026-04-02 追記: 15時入力の最初の解放ルール
- 15時を選んだときは、最初は15時列の「天気・気温・風」だけ入力可能。
- 15時列の3つを選び終えると、16〜21時の入力が解放される。
- 解放された時点で、16〜21時の初期値は15時の入力値でそろえる。

## pending / 後回しの巡回方針
- `少ない` を選んだエリアは、今までどおり後回し (`postponed_few`) を維持します。
- 手動スキップ (`skipped_manual`) も、今までどおり pending に残します。
- ただし pending を回る順番は、`manual` / `few` の理由切り替えよりも **現在位置からの近さ** を優先します。
- pending 中に同じエリアをもう一度 `少ない` / `スキップ` した場合は、そのエリアだけ一時的に後ろへ回します。
- `manual → few` / `few → manual` のような理由グループの強制切り替えは避け、揚げ物群と寿司群を往復しにくい挙動を優先します。

- 20時30分開始時は天候入力を行わない。
