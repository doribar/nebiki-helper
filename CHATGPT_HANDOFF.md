# ChatGPT 引き継ぎメモ

このファイルは、値引きヘルパー本体 ZIP に同梱するための引き継ぎ用プロンプトです。
次の ChatGPT セッションでは、ユーザーがこの ZIP をアップロードしたら、まずこのファイルを読んでから作業を開始してください。

## 最重要ルール
- この ZIP を、現時点の **値引きヘルパー本体の正本** として扱う。
- ZIP 名ではなく、ユーザーが次セッションに渡した ZIP の中身を正本として扱う。
- 同一セッション内で ChatGPT が新しい ZIP を生成した場合は、**最後に生成した ZIP** を最新の正本として扱う。
- ChatGPT が ZIP を更新して返すたびに、この `CHATGPT_HANDOFF.md` も必ず更新する。
- 次回セッションへの引き継ぎは、会話末尾で別途プロンプトを出すのではなく、ZIP 内のこのファイルを読む前提にする。

## 軽量 ZIP の方針
この本体 ZIP は、次のものを含めない方針です。

- `.git/`
- `node_modules/`
- `dist/`
- `.env`
- 実写真データ
- 個人情報・APIキー・秘密情報
- 一時ログや `npm_install.txt` / `tsc*.txt` / `check*.txt` などの作業ログ

依存関係は `package.json` と `package-lock.json` から復元します。

## 値引きヘルパー本体の主な仕様
- 15時 / 17時 / 18時30分 / 19時30分 / 20時30分の値引き運用を扱う。
- 曜日基準、天気、気温、風、雨・雪による補正を扱う。
- 20時30分の最終値引は `50 / 40 / 30` の3択固定。
- 20時30分開始時は天候入力を行わない。
- エリア順に巡回する。
- `少ない` と手動 `スキップ` は pending / 後回しとして扱う。
- pending の巡回順は、理由グループよりも現在位置からの近さを優先する。
- 戻る / 取り消しでは、画面状態だけでなく、次回値引記録や前回天気記録の復元も確認する。

## 保存まわりの方針
- 現在セッション、次回値引に影響する記録、前回天気記録は、できるだけ同じ状態の塊として扱う。
- `localStorage` を直接その場で増減させるのではなく、React state を更新してから保存する流れを優先する。
- 戻るや取り消しを直すときは、画面状態だけでなく、次回値引記録 / 前回天気記録まで巻き戻るかを確認する。
- `npm run check:logic` には、戻る履歴の積み方・復元スナップショット・保存記録の同梱確認も含めていく。

## 値引率補正の時間帯ルール
- 「曜日基準で補正しきれない分」の補正上限は時間帯で変わる。
- 強める側は、15時は `+5%` 上限、17時以降は `+10%` 上限。
- 緩める側は、15時は `-10%` 上限、17時以降は `-5%` 上限。
- ただし、30分〜1時間後の雨 `+10%` / 雪 `+20%` があるときは、この「補正しきれない分」は重ねない。
- 最終値引の点数計算では `±5% = ±1点`、`±10% = ±2点`。

## 時間別お天気入力
- StartScreen では、ウェザーニュースの1時間ごと表示を見ながら **15〜21時** の天気 / 気温 / 風速を入力する。
- 15時値引のときは、**15時と17時** の予報を比較する。
- 17時の気温が15時より **5度以上低い** ときは **曜日基準 +1段**。
- さらに、上の条件を満たしたうえで **17時の風速が15時より強い** ときは **さらに +1段**。
- この補正は通常の曜日基準補正に合流するので、「曜日基準で補正しきれない分」の +5% / +10% 判定にも影響する。
- 17時以降は、同じ時間別入力をそのまま参照するので、15時で入れた 18時予報は初期値として自然に引き継がれる。

## 15時入力の最初の解放ルール
- 15時を選んだときは、最初は15時列の「天気・気温・風」だけ入力可能。
- 15時列の3つを選び終えると、16〜21時の入力が解放される。
- 解放された時点で、16〜21時の初期値は15時の入力値でそろえる。

## StartScreen の初期値
- StartScreen の入力は、前回使った内容を初期値として引き継ぐ。
- ただし、手動固定していない曜日・時刻は、その時点の現在値に自動追従する。
- この挙動を触ったら、前回入力の保持とトップへ戻った直後の初期表示を確認する。

## 補正ラベルの方針
- 補正の内訳ラベルは、できるだけ短く同じ粒度でそろえる。
- 例: `気温 16〜20度 -1段` / `風 5m以上 +1段` / `近い雨 +10%` / `後の雪 +2段`
- 文章調の説明は要約や補足に寄せ、内訳はラベル列として読む前提にする。

## pending / 後回しの巡回方針
- `少ない` を選んだエリアは、今までどおり後回し (`postponed_few`) を維持する。
- 手動スキップ (`skipped_manual`) も、今までどおり pending に残す。
- pending を回る順番は、`manual` / `few` の理由切り替えよりも現在位置からの近さを優先する。
- pending 中に同じエリアをもう一度 `少ない` / `スキップ` した場合は、そのエリアだけ一時的に後ろへ回す。
- `manual → few` / `few → manual` のような理由グループの強制切り替えは避け、揚げ物群と寿司群を往復しにくい挙動を優先する。

## 写真判定サーバーについて
- 本体には写真判定連携コードがあるが、写真判定サーバー本体は別プロジェクトだった。
- 以前の `nebiki-photo-judge.zip` には `.env` が含まれていたため、APIキーを含む ZIP を共有しないこと。
- 写真判定サーバーを同梱・統合する場合も、`.env`、実写真、判定履歴、`node_modules` は含めない。

## 2026-05-01 追加修正: 写真判定フィードバック対象の厳密化
今回の修正で、写真判定サーバーを `nebiki-helper/nebiki-photo-judge/` として本体 ZIP 内に同梱した。

### 修正した問題
以前は、次回の人間判定を使って前回写真セットへフィードバックを付けるとき、条件が「同じエリア・過去の別時刻」程度だった。
そのため、理論上は `2026-05-01 15時` の写真セットに対するフィードバックを、別日や別の流れの判定で付けてしまう可能性があった。

### 修正後の仕様
写真セット保存時に、本体から写真判定サーバーへ `sessionDate` を送る。
写真判定サーバー側では `discountTimeKey` も保存する。
フィードバック対象は次の条件をすべて満たすものだけにした。

- 同じ `sessionDate`
- 同じ曜日基準
- 同じエリア
- 現在の値引時刻の直前時刻
  - 17時 → 15時
  - 18時30分 → 17時
  - 19時30分 → 18時30分
  - 20時30分 → 19時30分
- 人間判定済み
- まだ次回フィードバック未付与
- 現在の写真セットより古い

これにより、例えば `5月1日15時` のフィードバックは、原則として `5月1日17時` の同じエリア判定でだけ付く。
別日や直前でない値引時刻には付かない。

### 変更ファイル
- `src/domain/photoJudge.ts`
  - `/api/judge` 送信時に `sessionDate` を追加。
- `src/hooks/useNebikiApp.ts`
  - 写真判定キューと再送時に `state.session.date` を渡すように変更。
- `nebiki-photo-judge/src/server.js`
  - `sessionDate` / `discountTimeKey` を保存。
  - 前回フィードバック対象を「同じ日・同じ曜日基準・同じエリア・直前時刻」に限定。
  - 古いデータ向けに `createdAt` と `discountTime` からの後方互換フォールバックも残した。

## 2026-05-01 追加修正: 撮影時のメモリ不足対策
ユーザーから「メモリ不足がなんとかって出て写真撮るのがうまくいかなかった」と報告があったため、撮影後の画像処理を修正した。

### 修正した問題
高解像度写真を端末内で canvas 展開して圧縮する処理が、スマホ等のメモリの少ない環境で失敗し、撮影フロー自体が止まる可能性があった。
また、大きい元写真をプレビューとして表示すると、サムネイル表示でもブラウザが画像を展開してメモリを消費する可能性があった。

### 修正後の仕様
- `src/domain/photoJudge.ts`
  - 圧縮目標を軽めに変更: 長辺 `960 → 800 → 640`、目安 `450KB`。
  - 端末内圧縮に失敗してもエラーで止めず、元ファイルをそのまま返す。
  - これにより、メモリ不足で圧縮できない写真でも撮影済みとして扱える。
- `src/hooks/useNebikiApp.ts`
  - 1.2MB を超える写真はプレビュー用 `objectURL` を作らない。
  - 写真判定サーバーへのアップロードが成功したエリアは、保持していた `File` とプレビューURLを解放する。
  - UI上は撮影済み状態を維持するが、アップロード済みエリアの元写真はメモリから外す。
- `src/components/screens/PhotoCaptureScreen.tsx`
  - 撮影説明文を、メモリ不足時は元写真保持・大きい写真はプレビュー省略する内容に変更。
- `nebiki-photo-judge/src/server.js`
  - 端末側で圧縮できず元写真が送られる場合に備え、1枚あたりアップロード上限を `MAX_PHOTO_MB` で設定可能にした。
  - 既定値は `25MB`。
  - Multer のファイルサイズ/枚数エラーを分かりやすい日本語エラーで返す。
- `nebiki-photo-judge/.env.example`
  - `MAX_PHOTO_MB=25` を追加。

### 確認結果
- 本体: `npm run check:logic` PASS。主要チェック `41 / 41 checks passed`、追加 pending / 巡回チェックも PASS。
- 本体: `npm run build` PASS。
- 写真判定サーバー: `npm run check` PASS。

### 注意点
この修正は「圧縮時のメモリ不足で撮影が止まる」問題への対策。
ブラウザやOSのカメラ起動自体がメモリ不足で失敗する場合は、アプリ側の JavaScript まで制御が来ないため完全には防げない。その場合は、不要なタブを閉じる、端末を再起動する、写真なしで値引開始する、などの運用回避が必要。

## 2026-05-01 追加修正: 撮影フロー安定化

ユーザーから「なんか安定しない」と報告があったため、撮影直後に重い処理を行う構造を見直した。

### 背景
直前の修正では、撮影直後に以下を実行していた。

1. 端末内でアップロード用に圧縮
2. canvas でサムネイル作成
3. その後に撮影済みとして state へ保存

この方式は、スマホやメモリの少ない端末では、写真を撮った直後に画像デコード/canvas 展開が走るため不安定になりやすい。

### 今回の方針
撮影画面では「撮影済みとして即保存」を最優先にした。
重い圧縮処理は、撮影中ではなく「撮影を終えて値引開始」後の写真判定アップロード時に移動した。

### 修正内容
- `src/components/screens/PhotoCaptureScreen.tsx`
  - `compressPhotoForUpload` と `createPhotoPreviewUrl` を撮影直後に呼ばないようにした。
  - 撮影された `File` を即 `onCapturePhoto` へ渡す。
  - プレビューは軽い `URL.createObjectURL(file)` のみで作る。
  - プレビューURL作成に失敗しても撮影自体は保存し、「撮影済み」として進める。
  - `<img>` の読み込みに失敗した場合は画像表示をやめ、「撮影済み」プレースホルダーへ自動フォールバックする。
- `src/domain/photoJudge.ts`
  - canvas サムネイル作成関数をやめ、`createPhotoObjectPreviewUrl(file)` を追加。
  - `compressPhotoForUpload(file)` は残し、アップロード前の軽量化用として使う。
- `src/hooks/useNebikiApp.ts`
  - `requestPhotoJudgeForArea` 内で、写真判定サーバーへ送る直前に `compressPhotoForUpload` を順番に実行するようにした。
  - 圧縮に失敗しても `compressPhotoForUpload` は元ファイルを返すため、アップロード処理は止まりにくい。

### 重要な設計メモ
- 撮影画面では「撮影できたか」を最優先する。
- サムネイル表示は補助扱い。表示できなければ「撮影済み」プレースホルダーでよい。
- 写真判定AIへのアップロード・圧縮は値引開始後に裏で進める。
- 現場操作を止めないことを優先する。

### 次に同じ問題が出た場合
確認すべき観点:
- 撮影済みカウントが増えているか。
- サムネだけ出ないのか、撮影済み扱いにもならないのか。
- カメラアプリ/ブラウザ自体が落ちているのか。
- 特定端末・特定ブラウザだけで発生するか。

撮影済みカウントが増えているなら、サムネ表示だけの問題。
撮影済みカウントも増えないなら、`input[type=file]` の `onChange` が発火していない可能性がある。

## 2026-05-02 追加メモ: 過去の適切例を使ったAI判定方針

ユーザーと相談した結果、写真判定AIの将来方針は「今回写真を単独で多い/少ない/どちらでもない判定する」よりも、過去の適切だった例と今回写真を比較して判断する方向に寄せる。

### 最終的に目指す形
十分なデータが揃ったら、同じ曜日基準・同じ値引時刻・同じエリアの過去例から、次回フィードバックで適切だったと判断できる以下の3種類をAIに見せる。

- 過去の適切だった「多い」例
- 過去の適切だった「どちらでもない」例
- 過去の適切だった「少ない」例

そのうえで、今回写真が3つのうちどれに一番近いかをAIに判断させる。
比較時は、単なる画像の見た目の類似だけでなく、撮影角度、距離、エリア幅、商品の積み重なり方、空きスペース、商品密度、広く薄く残っているのか狭く厚く残っているのかも考慮させる。

### データが揃うまでの暫定方針
3種類の適切例が揃うまでは、同じ曜日基準・同じ値引時刻・同じエリアから、次回フィードバックで「適切だった」と扱える過去例を引っ張り、今回写真と比較させる。
このとき、過去例の判定は「どちらでもない」に限らない。過去例が「多い」でも「少ない」でも、次回値引で結果的に適切だったなら比較対象にしてよい。

暫定プロンプトの考え方:

```text
これは惣菜・弁当売場の値引き判断補助です。

過去の同じ曜日基準・同じ時刻・同じエリアでは、人間が「多い / どちらでもない / 少ない」のいずれかを選びました。
その後の次回値引で売れ残り状況が適正だったため、この過去判定は「適切だった」と扱っています。

今回の写真と過去写真を比較してください。
比較するときは、単に写真の見た目が似ているかだけでなく、撮影角度や距離、写真に写っている売場幅、商品の積み重なり方、空きスペース、商品密度、商品が広く薄く残っているのか狭い範囲に厚く残っているのかも考慮してください。

過去写真の判定を今回にも流用してよいかを判断し、必要なら「過去より多そう」「過去より少なそう」「判断が難しい」としてください。
```

### 過去の適切例を引っ張る件数
「件数」は写真1枚の数ではなく、過去の写真セット数を指す。
弁当・麺類エリアなら1件でも過去写真4枚セット、フライ・鶏惣菜なら1件につき過去写真2枚セットになる。

暫定の取得件数は以下にする。

- 弁当・麺類エリア: 1件
  - そもそも1セットあたりの写真枚数が多い。今回4枚 + 過去4枚だけでもAI入力が重くなるため。
- フライ・鶏惣菜エリア: 2件
  - 1セット2枚なので、2件見せても比較負荷が比較的軽い。
- それ以外のエリア: 3件
  - 1セットあたりの写真数が少ない想定なので、候補を増やして判断材料を補う。

該当する適切例が指定件数に満たない場合は、ある分だけ使う。
適切例がまったくない場合は、従来通り今回写真だけを見て参考判定するが、UI上は「過去の適切例なし」または「過去データ不足」と分かる表示にするのが望ましい。

### 実装時の注意
- 過去例の選定条件は、同じ曜日基準・同じ値引時刻・同じエリアを基本にする。
- 可能なら同じエリア内区分名、例: 正面 / バラ側 / 寿司側 / 麺類 / フライ側 / 鶏惣菜側、も写真メタデータとして扱い、同じ区分同士を比較しやすくする。
- AIへの説明では、過去例の人間判定と、その判定が次回フィードバックで適切だったことを明示する。
- 「過去例ベースの提案」と「AI単独の推測」を混同しない表示にする。
- 最終決定はこれまで通り人間のエリア判定ボタンで行い、その結果を次回フィードバックに回す。

### 現在のAI判定の動き
1. `/api/judge` で今回写真セットを保存する。
2. 保存済みデータから、同じ曜日基準・同じ時刻・同じエリアの「過去の適切例」を探す。
3. 過去適切例があれば、それらの写真セットと今回写真セットをAIへ渡す。
4. AIには、過去判定を今回にも流用できるか、または今回の方が多い/少ない方向に補正すべきかを判断させる。
5. 過去適切例がなければ、従来通り今回写真だけから控えめに参考判定する。
