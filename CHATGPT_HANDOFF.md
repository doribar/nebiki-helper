## 最重要ルール
- この ZIP の中身を、現時点の値引きヘルパー本体の正本として扱う。
- ZIP 名ではなく、ユーザーが次セッションに渡した ZIP の中身を正本として扱う。
- 同一セッション内で ChatGPT が新しい ZIP を生成した場合は、最後に生成した ZIP を最新の正本として扱う。
- ChatGPT が ZIP を更新して返すたびに、この `CHATGPT_HANDOFF.md` も必ず更新する。
- 会話上限対策として、引き継ぎは会話末尾の別プロンプトではなく、このファイルを読む前提にする。

## 祝日まわりの曜日基準ルール
今回、祝日判定を追加した。

### 追加ファイル
- `src/domain/japaneseHoliday.ts`

### 実装したルール
祝日・振替休日・国民の休日を日本の休日として扱う。祝日判定はアプリ内で計算する。

祝日当日の曜日基準は、天気・気温・風などの補正をかける前に、まず以下で決める。

- 祝日の15時: 日曜日扱い
- 祝日の17時以降で、翌日も休日・祝日なら: 金曜・土曜扱い
- 祝日の17時以降で、翌日が平日なら: 火曜・木曜扱い

ここでいう「翌日も休日・祝日」は、土曜・日曜・日本の祝日/振替休日/国民の休日を含む。

その後、通常どおり曜日基準補正をかける。

例:

- `2026-05-05 15時` → 祝日の15時なので日曜日基準
- `2026-05-05 17時` → 翌日 `2026-05-06` も振替休日なので金土基準
- `2026-01-12 17時` → 翌日が平日なので火木基準
- `2026-09-22 17時` → 国民の休日で翌日も秋分の日なので金土基準

### 注意
`src/domain/japaneseHoliday.ts` の春分・秋分は1980〜2099向けの近似式で計算している。通常運用年では実用上問題ない想定だが、将来の法改正や特別な祝日移動は別途反映が必要。

## 値引きヘルパー本体の主な仕様
- 15時 / 17時 / 18時30分 / 19時30分 / 20時30分の値引き運用を扱う。
- 曜日基準、祝日、天気、気温、風、雨・雪による補正を扱う。
- 20時30分の最終値引は `30 / 40 / 50%` の3択固定。
- 20時30分開始時は天候入力を行わない。
- エリア順に巡回する。
- `少ない` と手動 `スキップ` は pending / 後回しとして扱う。
- pending の巡回順は、理由グループよりも現在位置からの近さを優先する。
- 戻る / 取り消しでは、画面状態だけでなく、次回値引記録や前回天気記録の復元も確認する。

## 補正の基本方針
- 「曜日基準で補正しきれない分」の補正上限は時間帯で変わる。
- 強める側は、15時は `+5%` 上限、17時以降は `+10%` 上限。
- 緩める側は、15時は `-10%` 上限、17時以降は `-5%` 上限。
- ただし、30分〜1時間後の雨 `+10%` / 雪 `+20%` があるときは、この「補正しきれない分」は重ねない。
- 最終値引は固定表示。点数計算は現状使っていない。

## 時間別お天気入力
- StartScreen では、ウェザーニュースの1時間ごと表示を見ながら 15〜21時 の天気 / 気温 / 風速を入力する。
- 15時値引のときは、15時と17時の予報を比較する。
- 17時の気温が15時より5度以上低いときは曜日基準 +1段。
- さらに、上の条件を満たしたうえで17時の風速が15時より強いときはさらに +1段。
- この補正は通常の曜日基準補正に合流する。

## pending / 後回しの巡回方針
- `少ない` を選んだエリアは、後回し (`postponed_few`) を維持する。
- 手動スキップ (`skipped_manual`) も pending に残す。
- pending を回る順番は、`manual` / `few` の理由切り替えよりも現在位置からの近さを優先する。
- pending 中に同じエリアをもう一度 `少ない` / `スキップ` した場合は、そのエリアだけ一時的に後ろへ回す。

## 2026-05-04 追記：値引中の時刻経過ルール

今回、値引中に実時間が進んだ場合の扱いを追加・修正した。

## 2026-05-05 追記：エリア判定案内が一瞬で消える問題の修正

ユーザー報告：注意事項は表示されるが、弁当・麺類エリアのエリア判定案内（「まず多いかどうかを見る → 少ないかどうかを見る → どちらでもなければどちらでもない」）が、localStorage を消しても表示されない。

修正内容：

- `src/components/screens/AreaJudgeScreen.tsx` で `useRef` を使い、同じエリア表示中は `showJudgeGuide` が false に戻っても、すでに表示中の案内を消さないようにした。
- エリアが変わったときだけ `displayJudgeGuide` をそのエリアの `showJudgeGuide` に合わせてリセットする。
- これにより、弁当・麺類エリアでその日まだ案内を見ていない場合は、案内が画面に残ったまま見える。
- 表示済み記録は従来どおり `nebiki-helper/daily-message-state.bentoJudgeGuideShownDate` に保存される。

## 2026-05-05 追記：土日祝連休4日目以降の弁当系補正（実装済み）

ユーザーの実運用フィードバックより、土日祝が連続する長い連休では、連休終盤に弁当系が見た目以上に残りやすい可能性がある。

背景仮説:
- 連休中に外食・レジャー・買い物などで支出が増える
- 連休終盤は財布の紐が固くなりやすい
- 弁当は単価が高く、1食完結型になりやすい
- 客が「家で炊いたご飯 + おかずになる惣菜」に寄る可能性がある
- そのため、フライ・鶏惣菜などは悪くなくても、弁当・麺類系だけ残りやすい場合がある

現時点の方針:
- 連休3日目までは通常通りの値引ロジックでよい
- 連休4日目以降は、17時の値引から以下のエリアだけ `+10%` 上乗せするよう実装済み
  - 弁当・麺類
  - バランス弁当
- 15時から補正するかどうかは、今後の実績を見て判断する
- この連休補正は年1〜2回程度のレアイベント想定だが、ユーザー判断により実装済み

連休の定義:
- 土曜日・日曜日・祝日が連続している期間を連休として扱う
- 祝日は日本の祝日・振替休日・国民の休日を含む想定
- 例: 土・日・祝・祝 と続く場合、4日目の祝日から対象候補

実装済み条件:
```text
土日祝の連休日数 >= 4
かつ
値引時刻が17時以降
かつ
エリアが「弁当・麺類」または「バランス弁当」
→ 表示値引率に +10%
```

注意:
- 実装済み。
- 祝日連休の17時以降曜日基準ルールとは別の、エリア限定の追加補正案として扱う。
- 15時補正はまだ入れない。
